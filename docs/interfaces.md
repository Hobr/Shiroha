# Shiroha Interfaces

> 控制面接口、节点接口与外部交互边界

## 接口分层

Shiroha 对外分为两类接口:

- `Admin API`: 面向 CLI / Web / 运维系统的管理接口
- `Node RPC`: 面向 `Controller` 与 `Executor` 之间的内部接口

这两层接口的目标不同, 不应混用:

## 协议与传输层

### Admin API

- v0.1: 本地 HTTP/REST API，监听 Unix socket（默认）或 TCP loopback
- 定义为同步接口，简单直接，便于 CLI 和调试
- 后续版本可评估引入 gRPC-Web 或其他协议

### Node RPC

- v0.2: gRPC，以任务轮询 / 回执 / 心跳为主，保持与统一任务队列语义一致
- 传输层默认 TLS，节点注册时提供证书或共享 secret
- 节点间通信不经过 Admin API，避免内部接口泄露给外部调用方

- `Admin API` 负责注册制品、启动实例、查询状态、取消实例、发送 Signal、执行 Query
- `Node RPC` 负责节点注册、任务领取、心跳、回执、取消传播和健康检查

## Admin API

`Admin API` 至少需要覆盖以下对象与操作:

### Artifact / Workflow Definition

- 注册 Wasm 制品
- 创建或更新工作流定义版本
- 激活 / 废弃 / 禁用版本
- 查询制品与版本元数据

### Workflow Instance

- 启动实例
- 查询实例摘要与当前状态
- 查询实例事件历史
- 取消实例
- 可选的人工恢复或重新驱动

### Signal / Query

- 向实例发送 `Signal`
- 对实例执行 `Query`
- 返回明确的受理结果, 例如已接受、已去重、实例不存在、实例已终态

## Node RPC

`Node RPC` 只服务于控制面与执行面的协作, 至少需要覆盖:

- 节点注册
- 节点心跳
- 任务轮询 / 领取
- 租约续约
- 任务结果回执
- 取消传播
- 节点健康与能力声明同步

`Node RPC` 的设计应与统一任务队列语义保持一致:

- 节点通过轮询或等价机制获得待执行任务
- 任务执行期间通过 `Lease` / `heartbeat` 维持有效性
- 结果回执必须携带足够的任务身份信息, 以区分有效结果与 `Late Ack`

## Signal 语义

`Signal` 的执行语义（去重、排序、持久化约束）详见 [执行模型](execution-model.md)。此处仅描述接口层行为:

- 调用方提供 `signal_name`、载荷和可选 `dedup_key`
- 系统先持久化 Signal 事件, 再进入实例的正常排序与串行推进流程
- 返回明确的受理结果: 已接受、已去重、实例不存在、实例已终态
- 已终态实例默认拒绝新 Signal, 除非未来显式定义允许的特例

## Query 语义

`Query` 的执行语义详见 [执行模型](execution-model.md)。此处仅描述接口层行为:

- 只读接口, 不得修改事件历史、状态快照或实例状态
- 可以直接基于已提交快照回答, 或在 Wasm 内执行只读确定性查询入口
- 当实例不存在、制品未导出 Query 或查询名不存在时, 应返回明确错误

## 制品契约与接口边界

Wasm 制品与接口层之间至少需要约定以下导出边界:

- 必需导出: 决策入口、Activity 导出
- 可选导出: Query 导出
- 非目标: 通过接口层绕过事件历史或权限模型直接修改实例状态

这意味着:

- 决策与 Activity 是运行时契约的一部分
- Query 是否存在由制品显式声明
- Signal 永远通过控制面进入事件流, 而不是直接调用 Wasm 任意入口

## 错误模型

接口层建议统一暴露可判断的错误类别, 至少包括:

- 不存在: 实例、制品、版本、查询名或节点不存在
- 冲突: 版本状态冲突、实例状态冲突、重复注册
- 不兼容: WIT / ABI / 能力声明不兼容
- 无权限: 调用方权限不足或宿主能力不允许
- 失效: 任务 lease 已过期、结果为 `Late Ack`
- 不可执行: 当前无满足能力要求的节点

明确的错误类别比模糊字符串更重要, 因为 CLI、Web 和自动化系统都需要据此做稳定处理。
