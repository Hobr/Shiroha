# 数据流

描述一次事件从提交到完成的完整路径。相关语义不变量见 [semantics](05-semantics.md)。

## 事件入队

- `SubmitEvent` 到达主控后，事件进入对应实例的**串行队列**并分配 `event_id` 与 `event_seq`。
- 队列保证同一实例的事件严格按到达顺序处理；不同实例并发处理。
- 默认 ack 语义为 Applied (落库后返回)；可选 Accepted 语义。

## 一次事件的处理顺序

1. **事件提交**：客户端通过 `OrchestratorService.SubmitEvent` 向主控提交 `{instance_id, event}`。
2. **实例加载**：主控定位对应 FSM 实例，读取当前状态与绑定的 WASM 模块哈希。
3. **转换解析**：在 `shiroha-core` 中查出事件触发的转换与需要执行的 Action。
4. **策略读取**：从 Action 元数据读出 `DispatchSpec` 与 `AggregateSpec`。
5. **分发计划**：`Dispatcher` 根据已注册节点集合产出 `DispatchPlan`，并为每次调用分配 `attempt_id`。
6. **远程调用**：通过 `Transport` 并发调用目标节点；节点先 ack 再执行。
7. **节点执行**：节点在本地 WASM 模块缓存中查找；未命中则 `fetch_wasm` 拉取；随后在沙箱中执行 Action，受 fuel / 内存上限约束。
8. **结果聚合**：主控的 `Aggregator` 消费结果流，按声明策略产出最终 `ActionOutput`。
9. **状态落库**：状态迁移、事件日志、超时登记在同一 redb 事务中写入；失败则整体回滚。
10. **对外通知**：订阅者通过 `StreamEvents` 获取新状态；`SubmitEvent` 返回 ack。

## 失败与重试

- 节点执行失败或超时：按聚合策略判断是否补调；重发使用新的 `attempt_id` 的 retry_index。
- Transport 层失败：由其实现提供有限重试；彻底失败后向上层返回错误。
- 持久化失败：事务回滚，实例状态不变。
- 聚合失败：事件计入日志并标注 `aborted`；不触发状态迁移，不自动补偿。

## 超时事件注入

- 状态进入时若声明了停留时长约束，主控将超时记录登入**超时调度器** (与状态迁移同一事务)。
- 到期时超时调度器向实例队列注入一条系统事件，走正常事件处理流程。
- 主控重启后由 storage 恢复未到期的超时。

## 事件流订阅

- `StreamEvents` 接受游标 `from_seq`；主控按 `event_seq` 追加投递。
- 客户端断线重连后带上最后收到的 seq 即可续传。
- 事件日志只追加，历史不可变，便于重放与审计。

## WASM 分发

- 部署阶段由客户端上传 WASM 到主控，按内容哈希存入 `shiroha-storage`。
- 存入前主控校验导出世界与框架期望版本匹配；校验失败的 blob 不入库。
- 节点首次遇到未知哈希时反向拉取；其后缓存常驻。
- 哈希即版本；不同版本的 FSM 定义是不同实体，不做原地升级。
