# 安全基线

Shiroha 当前版本的安全模型为 **internal-trust**：假设部署网络受控，组件间默认不鉴权。本章列出生产启用前必须完成的项与长期目标。

## 传输加密

- 主控 ↔ 节点、主控 ↔ 客户端的 gRPC 通道建议 **mTLS**。
- 证书签发方案不由框架决定；`shiroha-config` 接受证书路径参数并透传到 tonic。
- 节点出站连接 (由 WASM 内的 `http-client` 发起) 受宿主策略约束，默认仅允许出站到显式白名单。

## 身份与授权

- **节点身份**: Register 时携带预共享凭据或客户端证书 CN；主控维护允许列表。
- **客户端身份**: Orchestrator 服务要求每次调用携带 token 或证书；身份用于审计日志。
- **授权粒度**: 首版按「全部允许 / 全部拒绝」；细化到实例级 ACL 放入远期规划。

## WASM 沙箱约束

- **Fuel / Epoch interruption**: 默认启用，限制单次 Action 最长执行时间。
- **内存上限**: 每个 Store 配置最大内存；超限触发 trap 并返回失败。
- **WASI capability 白名单**: 只开启 FSM 定义中声明过的能力；默认不给文件系统。
- **HTTP 出站白名单**: 通过 host 侧 WIT 实现，检查域名 / 方法 / 速率。

## 秘钥与敏感数据

- 主控的 redb 数据库文件按操作系统权限保护 (0600)；不做应用层加密 (留作远期)。
- 节点不持久化业务数据；WASM 模块缓存按内容哈希存放，不包含秘密。
- 日志与指标不得包含 Action payload 明文，仅记录大小与摘要。

## 生产前必须项

发布生产版本之前，至少完成：

1. mTLS 全链路启用。
2. 主控对客户端与节点的身份鉴别启用。
3. WASM fuel / 内存上限为默认配置，不能由用户覆盖为无限。
4. 审计日志接入外部系统，不只落本地。

这些项目不在 M0–M2 的开发范围内，但在文档里预留位置，避免落地时遗忘。
