# Shiroha v0.1 Performance Baseline

This is the first local reference baseline for the v0.1 warm path. It is a
regression comparison point, not a portable throughput or latency guarantee.

## Reference Environment

- Date: 2026-07-14
- CPU: Intel Core i7-11800H, 8 cores / 16 threads
- OS: Linux 7.1.3 x86_64
- Rust: 1.97.0 (`2d8144b78`, LLVM 22.1.6)
- Cargo: 1.97.0
- Wasmtime: 46.0.1
- wasm-tools: 1.253.0
- Profile: workspace `release` (`opt-level=3`, LTO, one codegen unit)
- Criterion: 20 samples, 0.5 second warm-up, 2 second measurement target

Commands:

```bash
cargo bench -p shiroha-core --bench core_engine -- \
  --warm-up-time 0.5 --measurement-time 2 --sample-size 20
cargo bench -p shiroha-adapter-wasm --bench wasm_runtime -- \
  --warm-up-time 0.5 --measurement-time 2 --sample-size 20
```

## Baseline

| Benchmark | 95% estimate | Point estimate |
|---|---:|---:|
| Host transition index lookup | 54.711–56.268 ns | 55.426 ns |
| Host startup plus one self-transition | 1.0557–1.0753 µs | 1.0650 µs |
| Component compile and full preparation | 28.480–29.331 ms | 28.910 ms |
| `InstancePre` instantiation | 15.955–16.108 µs | 16.027 µs |
| Warm typed guard call | 2.5428–2.6099 µs | 2.5758 µs |
| Warm typed callback call | 2.9699–3.0934 µs | 3.0282 µs |
| Warm typed action call | 3.5683–3.6551 µs | 3.6100 µs |

The compile/preparation result includes Component compilation, baseline WASI
linking, temporary instantiation, definition lifting/conversion, and Core
validation. It is intentionally separate from the warm guest call.

The callback/action rows were collected with the same 0.5 second warm-up and 20
samples using a 1 second measurement target. A repeated full run under an
interactive desktop workload showed wider warm-call variance, which is why the
policy below requires a reproduced regression rather than one measurement.

## Regression Policy

On the same reference class of machine and toolchain, investigate when:

- a Host or warm/instantiation point estimate regresses by more than 15%; or
- compile/preparation regresses by more than 20%.

A single noisy run does not fail a release. The regression must reproduce in a
second run with Criterion reporting a statistically significant change. Update
this baseline deliberately whenever the pinned toolchain, Wasmtime version,
Component fixture, or reference hardware changes.
